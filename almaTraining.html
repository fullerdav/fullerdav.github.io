Why Alma?

...previous systems were electronic representations of the card catalog
the electronic record looked like a card from the card catalog
it was much faster and easier to search
...this approach is ill-suited to electronic resources that lack a physical presence and change frequently
these resources now account for about 90% of our budget (or most of it anyway) and we need a better system to manage them
...the whole point of migration is to implement a system that serves our needs better than the old one
...migrations change everyone's jobs as well as every procedure related to the old system...
a migration is an opportunity to think about what we do and how we might do it better.

Alma is "inventory driven", what does that mean?

what we used to call the item record is most important
Alma inventory is distinguished by types, in our system there is physical (P) and electronic (E) inventory
actions that can be performed on a record are determined by it's type, for example:
[P]hysical inventory can be "received" and moved to a location
[E]lectronic inventory can be ordered and "activated", i.e., made available in discovery
inventory is grouped under holdings records, which also play a role in determining the inventory's availability for fulfillment
fulfillment, the act of delivering a resource to a patron, is also inventory driven, patrons want "their stuff"

Alma is inventory-driven and inventory is driven by requests

There are several request types ...we currently use "patron physical inventory" (aka pick from shelf), move (temporary or permanent) and work order requests
there are also purchase, booking, digitization, delivery none of which we currently use
requests are attached to inventory and action is required by staff with assigned roles, i.e., "pick from shelf" requires the request operator role
we currently use one type of work order request...does anyone know (or guess) what it is?

Processes in Alma

In Sierra we had statuses for missing, withdrawn, lost, etc. These are now processes.
There are onky two stauses in Alma, "Item in place", "Item not in place"
Item in place means on the shelf waiting to be borrowed
Item not in place means that an item is In Process, which might be "Loan", "Transit", "Missing", etc.
A process is more dynamic than a status and cannot always be easily updated, it might require running a "job"
jobs are an automated process that often involve checking various settings, tables, etc.

Creating Sets (logical or itemized)

sets of records can be saved and used for batch processing, updating, or simply exporting data
They are either logical or itemized, logical sets are saved queries which are dynamic, they search the repository each time you access them.
Itemized sets are fixed sets of records. They can be created from a logical set, imported from a file or from Alma Analytics, or stitched together manually
Sets are typically of the same record type, titles, holdings, po-lines, etc., in order to run jobs

Tools, when to use what?

Analytics is the easiest, most flexible reporting tool, but the data is always one day old. Also it does include every MARC field or allow us to combine data from disparate records easily.
Creating sets uses live data but it's output options are not very flexible. Best for running batch jobs.
Finally, exporting sets of MARC records in XML or MARC binary format can be done via as job. The output can then be processed in MARCedit or Excel.

Running Jobs

Alma id designed for efficiency, that includes batch processing, or jobs. jobs can be scheduled or run on demand. Some jobs are triggered automatically, after specific events, such as updating the calendar.
jobs depend on configured profiles, normalization rules and various tables and settings
jobs update records, processes, import and export records; they facilitate every major function, acqusitions, user management, etc.
naturally, the jobs you can run depend on your Alma roles
